# 캐치 티니핑 이름 맞추기 퀴즈 게임 기획/설계

## 1. 목표

- 나무위키 기반 캐릭터 정보를 수집해 로컬 JSON과 이미지 파일로 저장한다.
- 별도 서버 없이 HTML/CSS/JavaScript만으로 실행 가능한 정적 퀴즈 게임을 만든다.
- 사용자는 퀴즈 범위, 문항 수, 난이도를 선택한 뒤 바로 플레이할 수 있어야 한다.
- 모바일과 데스크톱 모두에서 보기 쉬운 캐주얼 퀴즈 UI를 제공한다.

## 2. 사용자 경험 목표

- 시작 화면에서 설정을 빠르게 끝내고 곧바로 퀴즈를 시작할 수 있어야 한다.
- 문제 화면은 이미지 중심으로 직관적이어야 한다.
- 정답/오답 결과는 한 번에 이해되고 감정 표현이 분명해야 한다.
- 전체 결과 화면은 성취감을 주고 재도전을 유도해야 한다.

- 시즌/기수
- 분류

## 5. 데이터 수집 방식 제안

### 5.1 권장 방식

- 1차: 티니핑 목록 페이지에서 시즌별 캐릭터 링크 수집
- 2차: 각 캐릭터 상세 페이지에서 대표 이미지와 설명 추출
- 3차: 로컬 JSON과 이미지 파일로 저장

## 6. 난이도 설계

### 6.1 하

- 인지도 높은 캐릭터 우선 출제
- 같은 시즌 내 대표 캐릭터 중심
- 보기 4개를 서로 성격이 다른 캐릭터로 구성
- 힌트가 직접적

### 6.2 중

- 같은 시즌/같은 분류 캐릭터를 섞어 출제
- 비슷한 스타일의 캐릭터를 보기로 포함
- 힌트가 간접적

### 6.3 상

- 외형이 비슷한 캐릭터끼리 보기 구성
- 동일 시즌 내 오답 선택지 비중 확대
- 변형 이름 캐릭터 포함
  - 예: 하츄핑 / 다이아 하츄핑 / 베리 하츄핑 / 스타 하츄핑
- 첫 힌트는 매우 제한적으로 제공

## 7. 퀴즈 생성 로직

### 7.1 출제 흐름

1. 사용자가 범위, 문항 수, 난이도 선택
2. 조건에 맞는 캐릭터 풀 생성
3. 중복 없이 문제용 캐릭터 샘플링
4. 각 정답 캐릭터마다 난이도에 맞는 오답 3개 선택
5. 보기 순서 랜덤 섞기

### 7.2 오답 선택 규칙

- 하: 시즌이 다르거나 이름이 명확히 다른 캐릭터 우선
- 중: 같은 시즌 또는 비슷한 분류 일부 포함
- 상: 같은 시즌, 같은 계열, 비슷한 이름, 동일 파생 캐릭터 우선

## 8. 힌트 설계

힌트는 최대 2개까지만 제공한다.

1. 힌트 1: 시즌, 분류, 모티브 중 하나
2. 힌트 2: 소품, 성격, 짧은 설명 중 하나

힌트는 버튼을 누르면 힌트 표시 영역에 한 줄 텍스트로 나타난다.

- `힌트 1 보기`
- `힌트 2 보기`
- `목소리 듣기`

목소리 버튼은 해당 캐릭터의 짧은 음성 클립이 있을 때만 활성화한다.

- 음성 클립이 있으면 재생
- 없으면 비활성화 또는 `준비 중` 안내

힌트를 많이 사용할수록 결과 화면에서 보너스 메시지를 조정할 수 있다.

- 캐주얼

- 카드형 인터페이스

### 9.2 화면 구성

- 상단 타이틀

## 10. 사운드/애니메이션 설계

### 10.1 정답

- 반짝이 파티클
- 스케일 업 효과
- 짧은 벨 또는 팡파르

### 10.2 오답

- 살짝 흔들리는 카드
- 작은 구름/한숨 효과
- 짧은 다운톤 효과음

### 10.3 구현 방식

- CSS 애니메이션 우선
- 사운드는 짧은 mp3 또는 브라우저 내장 Web Audio API 톤 생성
- 정적 배포 편의성을 위해 용량은 작게 유지

## 13. 예외 처리

- 선택 범위보다 문항 수가 많으면 자동으로 가능한 최대 수로 조정
- 이미지가 없는 캐릭터는 출제 제외 또는 대체 이미지 사용
- 설명이 너무 길면 요약본만 사용
- 데이터 수집 실패 시 이전 캐시 데이터 유지

- 데이터 검수

## 16. 캐주얼 퀴즈 게임 확장 아이디어

- 연속 정답 콤보 시스템
- 힌트 적게 쓰면 별 3개 평가
- 시즌별 도감 해금
- 오답노트 모드
- 시간 제한 모드
- 하루 10문제 챌린지
- 부모/아이 함께하기 모드
- 틀린 문제 다시 풀기
- 캐릭터 실루엣 모드
- 목소리/설명 듣기 모드

## 17. 리스크 및 체크 포인트

- 나무위키/미러의 문서 구조가 일정하지 않을 수 있다.
- 이미지 링크 형식이 페이지마다 다를 수 있다.
- 저작권/이용 조건 관점에서 원문 전체 복제보다 게임용 최소 정보 저장이 안전하다.
- 장기적으로는 수집 스크립트와 결과 JSON을 분리해 재생성 가능하게 관리하는 것이 좋다.
